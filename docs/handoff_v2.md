## 1. プロジェクト概要

- アプリ名: **CareStrings**。由来=入居者や家族の思いが「弦(strings)」となって職員に共有され繋がるイメージ。README・要件定義書§1.4に記載済み。
- 特養(特別養護老人ホーム)向けの申し送り・情報共有Webアプリ。**転職用ポートフォリオ**として個人開発する。
- 志望企業: **ALTURA X株式会社**(医療SaaSスタートアップ、健診DX×PHR、HL7 FHIR活用)。利用技術スタック(確認済み): **Go / TypeScript / React / React Native / AWS / MySQL / Redis / GraphQL / gRPC**。求人業務内容に「GoによるAPI開発(標準ライブラリ中心、GraphQLスキーマ設計含む)」、歓迎条件に「GoによるGraphQL・gRPC実装経験」。ジュニア枠には「Go未経験歓迎」あり。
- 開発者スキル: VB.NET 1年、C# 約1.5年、HTML/CSS/JS 3〜4年。**Goは未経験**。AIをペアプロ相手に学びながら作る方針(隠さず「AI活用+Go初挑戦」とREADMEに明記する戦略で合意済み)。

## 2. 原体験と課題(このプロジェクトの核)

- 開発者の**母が要介護5で特養に入居中**。母の不満:「職員に伝えたことが他の職員に共有されていない。何度も同じことを言わされる」= ケアを受ける側から見た情報共有の失敗。
- 特に外国人職員への伝達が不確実。身近に「**日本語の会話はできるが漢字が読めない**」外国人職員が実在する。
- 業界調査でも裏付けあり: 情報共有の課題1位「申し送りノートの不明点確認」、2位「引き継ぎ」。原因「内容を確認しない人がいる」「解釈の違い」「手書きで履歴が残らず言った言わないになる」(ケアきょうのアンケート)。
- 施設の職員に**ヒアリング可能**(母の入居先の職員と話せる)。

## 3. 確定した技術構成

| 領域 | 技術 |
|---|---|
| フロント | Next.js (App Router) + TypeScript |
| バックエンド | Go (Echo or Gin) の REST API ※フェーズ2で移行 |
| DB/認証/画像 | Supabase (PostgreSQL + Auth + Storage) |
| ふりがな | kuroshiro(クライアント側生成、DB保存しない) |
| 翻訳 | DeepL API等(投稿時に翻訳→DBキャッシュ方式) |
| AI要約 | Claude API |
| UI多言語 | next-intl(i18n辞書) |
| ホスティング | Vercel(フロント)+ Cloud Run/Railway(Go) |
| PWA | 後から追加(最後でよい) |

- 開発順序: **フェーズ1= Next.js+Supabase直結でMVP完成・公開 → フェーズ2= Go(標準net/http中心)でAPI層を構築。最初の2〜3エンドポイントはRESTで肩慣らし後、gqlgenでGraphQL APIとして本実装**(志望先のスタックにGraphQL明記のため計画に昇格。投稿詳細=投稿+著者+翻訳+確認状況+コメントの1クエリ取得がスキーマ設計の見せ場)。コミット履歴に過程を残す。
- AWSは使わない(無料枠と開発速度優先。面接では「小規模個人開発での費用対効果判断」と説明)。gRPCは見送り(この規模で使うと不自然。ロードマップに「サービス分割時に検討」と記載する程度)。

## 4. 主要な設計判断(合意済み)

1. **既読と確認を分離**: 既読=詳細を開いたら自動記録(read_at)/確認=「内容を確認しました」ボタン(confirmed_at)。post_receiptsテーブル(投稿×職員で1行)。重要投稿は確認必須で、ホーム上部バナーに確認まで滞留。
2. **表示モード3択**: 日本語 / **ふりがな付き日本語(本命。漢字が読めない職員向け)** / 翻訳(英・タイ語)。UIメニューも投稿内容も同じモード設定で切替。実装順: 投稿ふりがな→UI辞書→翻訳API。
3. **翻訳は投稿時キャッシュ方式**(表示のたびにAPIを叩かない)。UIメニューはi18n辞書(コストゼロ)。ふりがなはkuroshiroでクライアント生成(DBに持たない)。
4. **バイタルは「異常値→申し送り投稿を提案」で共有と接続**するのが目玉。基準値はハードコードせず施設設定(vital_thresholds)。
5. **フロー/ストックの分離**: 申し送り(posts=流れる)と恒常的な注意事項(resident_notes=利用者詳細に常時表示、カテゴリ別)を分けた。
6. **コメント機能あり(フラット・ネストなし)**: 現場課題1位「申し送りの不明点確認」への解答。投稿・コメントとも**編集不可**(「言った言わない」防止。訂正は新規投稿・コメントで)。
7. **通知2ルール**: ①自分の投稿にコメント ②自分が参加したコメント欄に新着 → 質疑ループが閉じる。ベルバッジ+通知一覧。
8. **管理者専用の登録**: 職員(招待メール)・利用者の登録/編集はadminのみ。UI出し分け+Supabase RLSの2層防御。
9. **棟タブ+絞り込みフィルタ**: ホームは棟(1棟/2棟/全体)タブ×(すべて/利用者宛/施設連絡)フィルタ。施設連絡=resident_id NULL に後から辿り着ける。
10. **投稿カードに投稿者のアイコン+名前+役職チップ**(所属棟は非表示=タブで自明)。ホームとダッシュボードは1画面統合(重要バナー+AI要約+タイムライン)。
11. 削除でなくis_activeフラグ(履歴保全)。全テーブルにupdated_at(トリガー自動更新)。命名規約=小文字スネークケース・複数形(一般的と確認済み)。

## 5. 法務・データ方針(重要、合意済み)

- **実在の入居者・職員の情報は一切入れない**。介護・健康情報は要配慮個人情報。名前だけ変えた仮名化でも照合可能なら個人情報のままなのでNG。
- 検証は**完全架空の利用者・シナリオ**を作り、職員に操作してもらい感想を聞く**ユーザビリティテスト**形式で実施。
- READMEに「実運用には医療情報ガイドライン準拠が必要。本アプリはデモ用途」と明記する。

## 6. 競合と差別化

- 既存: ケアコラボ(利用者軸タイムライン、申し送り時間短縮の実績)、ほのぼの、カイポケ等多数。
- 差別化: **ふりがな/多言語ネイティブ対応 + AI要約 + 既読/確認の分離**。ケアコラボは調べた範囲で多言語・AI要約を製品内蔵していない。
- ビジネスとして成功は困難(レッドオーシャン、施設のIT予算薄)だが、ポートフォリオ・転職の道具としては有望、という評価で合意。
