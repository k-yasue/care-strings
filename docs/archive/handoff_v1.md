## 1. プロジェクト概要

- 特養(特別養護老人ホーム)向けの申し送り・情報共有Webアプリ。**転職用ポートフォリオ**として個人開発する。
- 志望企業: **ALTURA X株式会社**(医療SaaSスタートアップ、健診DX×PHR、HL7 FHIR活用、約40人)。技術スタックは**バックエンドGo / フロントTypeScript・React**。求人に「Go未経験歓迎」あり。
- 開発者スキル: VB.NET 1年、C# 約1.5年、HTML/CSS/JS 3〜4年。**Goは未経験**。AIをペアプロ相手に学びながら作る方針(隠さず「AI活用+Go初挑戦」とREADMEに明記する戦略で合意済み)。

## 2. 原体験と課題(このプロジェクトの核)

- 開発者の**母が要介護5で特養に入居中**。母の不満:「職員に伝えたことが他の職員に共有されていない。何度も同じことを言わされる」= ケアを受ける側から見た情報共有の失敗。
- 特に外国人職員への伝達が不確実。身近に「**日本語の会話はできるが漢字が読めない**」外国人職員が実在する。
- 業界調査でも裏付けあり: 情報共有の課題1位「申し送りノートの不明点確認」、2位「引き継ぎ」。原因「内容を確認しない人がいる」「解釈の違い」「手書きで履歴が残らず言った言わないになる」(ケアきょうのアンケート)。
- 施設の職員に**ヒアリング可能**(母の入居先の職員と話せる)。

## 3. 確定した技術構成

| 領域 | 技術 |
|---|---|
| フロント | Next.js (App Router) + TypeScript |
| バックエンド | Go (Echo or Gin) の REST API ※フェーズ2で移行 |
| DB/認証/画像 | Supabase (PostgreSQL + Auth + Storage) |
| ふりがな | kuroshiro(クライアント側生成、DB保存しない) |
| 翻訳 | DeepL API等(投稿時に翻訳→DBキャッシュ方式) |
| AI要約 | Claude API |
| UI多言語 | next-intl(i18n辞書) |
| ホスティング | Vercel(フロント)+ Cloud Run/Railway(Go) |
| PWA | 後から追加(最後でよい) |

- 開発順序: **フェーズ1= Next.js+Supabase直結でMVP完成・公開 → フェーズ2= API層をGoに移行**(コミット履歴に過程を残す)。
- AWS・GraphQL・gRPCは**使わない**(オーバーエンジニアリング回避。求人の必須でもない)。

## 4. 主要な設計判断(合意済み)

1. **既読と確認を分離**: 既読=詳細を開いたら自動記録(read_at)/確認=「内容を確認しました」ボタン(confirmed_at)。重要投稿は確認必須で、ホーム上部バナーに確認まで滞留。「読んだけど確認してない人」を可視化。
2. **表示モード3択**: 日本語 / **ふりがな付き日本語(本命。漢字が読めない職員向け)** / 翻訳(英・タイ語)。UIメニューも投稿内容も同じモード設定で切替。実装順: 投稿ふりがな→UI辞書→翻訳API。
3. **翻訳は投稿時キャッシュ方式**(表示のたびにAPIを叩かない。コストと速度の両立)。UIメニューはi18n辞書ファイル(コストゼロ)。この2層の使い分けが設計の見せ場。
4. **バイタルは残す**。単体記録でなく「異常値→申し送り投稿を提案」で記録と共有を接続するのが目玉。基準値はハードコードせず**施設ごとに設定変更可能**(ドメイン知識の限界を設計でカバー)。
5. **投稿カードに投稿者のアイコン+名前+役職チップ**を表示(看護師/介護士で読み手の受け取りが変わるため)。所属棟はカードに出さない(棟タブで自明)。
6. **棟(1棟/2棟/全体)タブ**: 職員は所属棟を持ち、ログイン時は自分の棟がデフォルト表示。閲覧制限は基本なし。
7. **ホームとダッシュボードは1画面に統合**: 重要バナー+AI要約(本日の要点)+タイムライン。
8. 削除でなく is_active フラグ(履歴保全)。全体連絡は resident_id=NULL で投稿テーブル一本化。

## 5. 法務・データ方針(重要、合意済み)

- **実在の入居者・職員の情報は一切入れない**。介護・健康情報は要配慮個人情報。名前だけ変えた仮名化でも照合可能なら個人情報のままなのでNG。
- 検証は**完全架空の利用者・シナリオ**を作り、職員に操作してもらい感想を聞く**ユーザビリティテスト**形式で実施。
- READMEに「実運用には医療情報ガイドライン準拠が必要。本アプリはデモ用途」と明記する。

## 6. 競合と差別化

- 既存: ケアコラボ(利用者軸タイムライン、申し送り時間短縮の実績)、ほのぼの、カイポケ等多数。
- 差別化: **ふりがな/多言語ネイティブ対応 + AI要約 + 既読/確認の分離**。ケアコラボは調べた範囲で多言語・AI要約を製品内蔵していない。
- ビジネスとして成功は困難(レッドオーシャン、施設のIT予算薄)だが、ポートフォリオ・転職の道具としては有望、という評価で合意。
