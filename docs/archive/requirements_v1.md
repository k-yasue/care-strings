# 要件定義書 — 介護施設向け申し送り共有アプリ(仮称: ケアシェア)

- 版: v1.0
- 作成日: 2026-07-14
- 種別: 個人開発 / 転職用ポートフォリオ
- 関連資料: `wireframe_v2.html`(画面設計)

---

## 1. 背景

### 1.1 きっかけ(原体験)

開発者の母は特別養護老人ホームに要介護5で入居している。母から繰り返し聞く不満が本プロジェクトの出発点である。

> 「職員さんに伝えたことが、他の職員さんに共有されていない。何度も同じことを言わないといけない。」

これはケアを受ける側から見た「情報共有の失敗」である。特に外国人職員に対しては伝達がさらに不確実になっているとの実感があった。

### 1.2 業界調査による裏付け

介護職員向けアンケート(ケアきょう調べ)では、情報共有の課題を感じるシーンの上位が「申し送りノートの不明点確認・補足」「引き継ぎ」であり、原因として「申し送りノートの内容を確認しない人がいる」「職員ごとの解釈の違い」「手書きで履歴が残らず水掛け論になる」等が挙げられている。

紙の申し送りノートは記載者によって情報量・構成がバラつき、その場に行かないと読めないという構造的な問題も指摘されている。

また、外国人介護人材は国の受け入れ拡大方針により増加傾向にあるが、「日本語での会話はできるが漢字の読みが困難」という職員が存在し、文字ベースの申し送りが機能しにくい実態がある(開発者の身近にも該当者あり)。

### 1.3 解決したい課題(要約)

1. 申し送りの情報量・書き方が職員によってバラバラ
2. 申し送りを「読んだかどうか」「理解・了承したかどうか」が分からない
3. 漢字が読めない外国人職員に申し送りが届いていない
4. 入居者・家族が伝えた内容が職員間で共有されない

## 2. 目的

### 2.1 プロダクトとしての目的

特養等の介護施設内の申し送りを、「書く・読む・確認する」まで確実に完結させるモバイルファーストのWebアプリを提供する。

### 2.2 ポートフォリオとしての目的

- モダンな技術スタック(Next.js / TypeScript / Go / Supabase)での設計・実装力の証明
- 実在の課題に対するヒアリング → 要件定義 → 設計 → 実装 → ユーザビリティテストという一連の開発プロセスの証明
- LLM API(要約・翻訳)を組み込んだAI活用開発の経験獲得
- 応募先(医療データ連携領域の企業)の事業ドメインと重なる題材選定

## 3. ユーザーと利用シーン

### 3.1 想定ユーザー(ペルソナ)

| ペルソナ | 状況 | 主なニーズ |
|---|---|---|
| 日本人介護士(20〜50代) | 多忙。申し送りは業務の合間に書く/読む | 短時間で漏れなく書ける・読める |
| 外国人介護士 | 日本語会話は可能だが漢字の読みが困難 | ふりがな or 母語で内容を理解したい |
| 看護師 | 医療的視点の申し送りを行う | 自分の申し送りが確実に全員へ届くこと |
| 施設管理者(リーダー) | 伝達徹底の責任者 | 誰が未確認かを把握し、フォローしたい |

### 3.2 利用環境

- 主端末: 職員個人のスマートフォン、施設共用タブレット
- スマホファースト(将来PWA化)

## 4. 機能要件

### 4.1 MVP(第1リリース)

| # | 機能 | 概要 |
|---|---|---|
| F-01 | 認証 | メール+パスワードによるログイン。アカウントは管理者が発行(自由登録なし) |
| F-02 | 職員プロフィール | 氏名・アイコン・役職(介護士/看護師/管理者)・所属棟・表示言語モード |
| F-03 | 利用者管理 | 氏名・部屋番号・所属棟・要介護度・注意事項(架空データのみ) |
| F-04 | 申し送り投稿 | 対象利用者(または全体連絡)・宛先棟・カテゴリ・本文・写真・重要フラグ・家族了承フラグ |
| F-05 | 入力テンプレート | カテゴリ別の定型項目を本文に挿入し、情報量の個人差を抑制 |
| F-06 | タイムライン | 棟タブ(所属棟/他棟/全体)切替。新着順・無限スクロール。投稿者(名前・アイコン・役職)と宛先利用者を表示 |
| F-07 | 重要バナー | 重要フラグ付き・未確認の投稿をホーム上部に確認まで滞留表示 |
| F-08 | 既読/確認の分離 | 既読=詳細を開いた時点で自動記録。確認=「内容を確認しました」ボタンの明示的押下。重要投稿は確認必須 |
| F-09 | 確認状況の可視化 | 投稿ごとに確認済/既読のみ/未読の職員一覧。個人ページに自分の未確認リスト |
| F-10 | 利用者別タイムライン | 利用者詳細画面に当該利用者宛の投稿が時系列で蓄積。カテゴリ絞り込み |
| F-11 | ふりがなモード | 投稿本文に自動でふりがなを付与して表示(kuroshiro)。漢字が読めない職員向けの主要機能 |
| F-12 | バイタル記録 | 体温・血圧・脈拍・SpO2の入力と履歴・グラフ表示 |
| F-13 | 異常値→申し送り接続 | 基準値(施設ごとに設定変更可能)を外れた場合に警告し、値を引き継いだ申し送り投稿を提案。記録と共有の分断を防ぐ |

### 4.2 第2リリース(差別化機能)

| # | 機能 | 概要 |
|---|---|---|
| F-14 | 翻訳モード | 投稿本文を英語・タイ語等へ翻訳表示。投稿時に翻訳しDBへキャッシュ(コストと表示速度の両立)。外国語で書かれた投稿の日本語化にも対応 |
| F-15 | UI多言語化(i18n) | メニュー等の固定文言を辞書ファイルで切替(日本語/ふりがな/英語)。表示モード設定は投稿・UIで共通 |
| F-16 | AI要約 | 当日の重要・新着投稿から「本日の要点」を生成しホームに表示(LLM API)。言語モードに追従 |
| F-17 | PWA化 | ホーム画面追加・プッシュ通知(未確認の重要投稿) |

### 4.3 表示モードの仕様

職員設定で以下の3モードを選択。投稿内容とUI文言の双方に適用される。

1. 日本語(標準)
2. ふりがな付き日本語 — 「会話はできるが漢字が読めない」職員向けの本命機能。ヒアリングに基づく設計判断
3. 翻訳(英語 / タイ語) — 来日直後等、日本語読解が難しい職員向け

実装順序: 投稿のふりがな → UI辞書(ふりがな/英語) → 投稿の翻訳API。

### 4.4 やらないこと(スコープ外)

- シフト管理、介護保険請求、ケアプラン作成、家族向けアカウント
- 医療機器連携、帳票出力
- 実在の入居者・職員の個人情報の取り扱い(§6参照)
- 医学的判断の自動化(バイタル基準値はデフォルト参考値+施設設定制とし、判断は現場に委ねる)

## 5. 非機能要件

| 区分 | 要件 |
|---|---|
| 性能 | タイムラインはページネーション(無限スクロール)で取得。投稿日時・利用者ID等にインデックスを設定し、データ増加時も応答速度を維持 |
| ストレージ | 画像はアップロード前にクライアント側で圧縮・リサイズ(browser-image-compression想定)。タイムラインはサムネイル表示 |
| コスト | 翻訳・要約は生成時1回のみAPI実行しDBへキャッシュ。表示時のAPI呼び出しは行わない |
| セキュリティ | 全通信TLS。認証必須。Supabase RLS(Row Level Security)によるアクセス制御 |
| 対応環境 | モバイルブラウザ(iOS Safari / Android Chrome)最優先。PC対応 |

## 6. 個人情報の取り扱い方針

- 本アプリはデモ・検証用途であり、実在の入居者・職員の情報は一切入力しない。利用者データはすべて架空
- 仮名化(名前のみ変更し照合可能な状態)でも個人情報保護法上の個人情報・要配慮個人情報に該当し得るため、実データの投入は行わない
- 現役介護職員によるユーザビリティテストは、架空の利用者・シナリオを用いた操作体験と感想聴取の形式で実施する
- 実運用には医療情報システムの安全管理に関するガイドライン等への準拠が必要であることを認識した上で、本プロジェクトではスコープ外とする

## 7. 技術構成と選定理由

| 領域 | 技術 | 選定理由 |
|---|---|---|
| フロントエンド | Next.js (App Router) + TypeScript | 開発者のJS/TS経験を活かせる。求人市場・応募先スタックとの一致 |
| バックエンド | Go (Echo または Gin) + REST API | 応募先の主要言語。未経験だがAIペアプログラミングを活用し学習しながら実装(READMEに明記) |
| DB / 認証 | Supabase (PostgreSQL + Auth + Storage) | 無料枠で認証・DB・画像ストレージ・リアルタイム更新が完結 |
| ふりがな | kuroshiro | API不要・無料で形態素解析ベースのふりがな付与が可能 |
| 翻訳 | DeepL API ほか(タイ語対応可否で選定) | 無料枠内で投稿時キャッシュ方式により運用 |
| AI要約 | Claude API | LLM活用開発の経験獲得 |
| i18n | next-intl | Next.js での多言語UIの定番 |
| ホスティング | Vercel(フロント)+ Cloud Run または Railway(Go API) | 無料〜低コストで公開可能 |

段階計画: フェーズ1はNext.js + Supabase直結でMVPを完成・公開し、フェーズ2でAPI層をGoへ移行する。移行過程をコミット履歴として残す。

## 9. 検証計画(ユーザビリティテスト)

- 対象: 母が入居する特養の職員(可能なら日本人・外国人各1名以上)
- 方法: 架空利用者・架空シナリオ(例: 口腔ケア用品の交換頻度変更、家族了承済み)を用意し、投稿→閲覧→確認の一連操作を体験してもらう
- 聴取項目(案):
  - 現場のスピード感で入力できるか / テンプレート項目は実態に合うか
  - ふりがな表示は読みやすいか(外国人職員)。翻訳とどちらが有用か
  - 既読と確認の分離は現場感覚に合うか
  - 現状の申し送り(紙・口頭)と比べた率直な感想
- 結果はREADMEに「ヒアリングと改善内容」として記録する

## 11. 用語集

| 用語 | 意味 |
|---|---|
| 申し送り | シフト交代時等に行う職員間の情報引き継ぎ |
| 特養 | 特別養護老人ホーム。要介護高齢者向けの入所施設 |
| バイタル | バイタルサイン。体温・血圧・脈拍・SpO2等の生命兆候 |
| 既読 / 確認 | 既読=投稿詳細を開いた記録(自動)。確認=内容の理解・了承をボタンで明示した記録 |
| i18n | 国際化。UI固定文言を辞書ファイルで多言語切替する仕組み |
| PWA | ホーム画面追加・プッシュ通知等が可能なWebアプリ形式 |
